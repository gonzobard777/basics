## Правила против 100% загрузки диска при воркфлоу (на этом WSL-окружении)

```
у тебя постоянная проблема при генерации задания для агентов воркфлоу -> ``
не используй `` в задании воркфлоу

1. Потолок одновременных агентов — не больше 4 (вместо дефолтных ~16). 
   Причина падения была в пике памяти от множества параллельных Opus-контекстов → своп в pagefile на C: → 100% I/O. 
   Меньше агентов разом = ограниченная память и I/O.
2. Никаких широких/рекурсивных сканов ФС по /mnt/d из агентов. 
   Перечисление файлов (find, grep -r) делаю один раз сам в основном цикле и передаю агентам готовые абсолютные пути. 
   Агентам — только Read по конкретным путям и grep, ограниченный одной известной подпапкой. Мост DrvFs/9p к диску Windows — главный генератор I/O-нагрузки.
   Если же агенту переданы не все пути к требуемым ему файлам, то допускается, чтобы агенты могли самостоятельно читать из файловой системы, только бережно.
3. Не держать много «тяжёлых» (Opus/xhigh) генераций одновременно. 
   Максимум один-два разом, остальное — последовательно (pipeline, а не широкий барьер). 
   Большой контекст × много агентов = тот самый пик RAM.
```

## Градация сложности вопросов (1-5)

1 — Тривиально: факт, синтаксис, очевидная ошибка. Отвечай сразу.
2 — Просто: небольшой анализ, стандартный паттерн. Минимум раздумий.
3 — Средне: несколько факторов, требует обдумывания.
4 — Сложно: архитектура, неочевидные зависимости, глубокий анализ.
5 — Максимально: системные решения, сложные баги, критические компромиссы. Используй extended thinking.

## Как работать с Codex на длинных задачах

- Храни ключевые вводные в файле проекта, а не только в чате.
- Для длинной задачи заведи один короткий файл состояния: цель, ограничения, текущее решение, открытые вопросы.
- Перед новой подзадачей давай ссылку на этот файл или путь к нему.
- Важные инварианты формулируй явно и коротко, чтобы их можно было быстро перечитать.
- После крупных изменений фиксируй промежуточный итог в 5-10 строках.
- Не растягивай одну сессию на слишком много несвязанных тем.
- Если тема сильно ушла в сторону, лучше начать новую сессию с краткого резюме.
- Если есть критичные места в коде, указывай конкретные файлы и методы, а не описывай их только словами.
- Если важно ничего не забыть, проси меня заново перечитать нужные файлы перед выводом или изменением, как ты уже и делаешь.
- Самый надежный источник истины для длинной работы: файлы проекта, а не накопленная история диалога.

Когда задача становится длинной, запутанной или критичной, ты можешь попросить меня подготовить такой документ по текущему состоянию задачи:

- собери context.md по текущей задаче
- подготовь файл контекста по этому шаблону
- зафиксируй текущее состояние задачи в context.md
- обнови context.md после последних изменений
